## ¿Qué ventajas le proporciona a un proyecto el uso de un pipeline de CI? Menciona al menos tres ventajas específicas y explica por qué son importantes.

El uso de pipelines de integración continua le aporta muchas ventajas a un proyecto, especialmente porque ayuda a mantener la calidad del código desde etapas tempranas. Una de las ventajas más importantes es que permite detectar errores rápidamente, ya que cada cambio que se sube al repositorio pasa automáticamente por pruebas y validaciones. Así, si algo falla, se identifica de inmediato y no cuando el problema ya es más grande. También ayuda a automatizar tareas repetitivas como instalar dependencias, correr pruebas o construir la aplicación, lo cual ahorra tiempo y reduce errores manuales. Además, da más confianza al equipo al momento de integrar cambios, porque todos saben que el código fue verificado antes de continuar, lo que hace que el proyecto sea más estable y que las entregas sean más seguras.

## ¿Cuál es la diferencia principal entre una prueba unitaria y una prueba de aceptación? Da un ejemplo de algo que probarías con una prueba unitaria y algo que probarías con una prueba de aceptación (en el contexto de cualquier aplicación que conozcas, descríbela primero).

La principal diferencia entre una prueba unitaria y una prueba de aceptación está en el alcance de lo que se prueba. Una prueba unitaria se enfoca en verificar que una pequeña parte del código funcione correctamente, como una función o un método aislado. por ejemplo, en una tienda virtual, podría probarse que la función que calcula el total de una compra sume correctamente los precios de los productos. En cambio, una prueba de aceptación valida que una funcionalidad completa cumpla con lo que espera el usuario final, siguiendo el mismo ejemplo, se podría probar que un usuario pueda agregar productos al carrito, realizar el pago y recibir una confirmación exitosa. En resumen, la prueba unitaria revisa partes específicas del sistema, mientras que la prueba de aceptación evalúa si el sistema realmente cumple con el comportamiento esperado desde la perspectiva del usuario.

## Describe brevemente qué hace cada uno de los steps principales de tu workflow de GitHub Actions (desde el checkout hasta el push de Docker). Explica el propósito de cada uno (qué hace y para qué se hace).

En el workflow de GitHub Actions, el proceso comienza con el paso de checkout, que se encarga de descargar el código fuente del repositorio para que el pipeline pueda trabajar sobre la versión más reciente del proyecto. Luego, en Set up Python, se configura el entorno instalando la versión específica de Python requerida por la aplicación, asegurando que la ejecución sea consistente en cada corrida del pipeline. Después, en Install dependencies, se actualiza pip y se instalan las dependencias definidas en el archivo requirements.txt, preparando así el entorno para ejecutar las validaciones y pruebas necesarias. Posteriormente se ejecutan herramientas de análisis estático como Black, que verifica que el formato del código cumpla con el estándar establecido, y Pylint y Flake8, que analizan la calidad del código y detectan posibles errores o problemas de estilo, ayudando a mantener un código más limpio y ordenado. Luego se realizan las pruebas unitarias con pytest, las cuales verifican que las funciones internas de la aplicación se comporten correctamente, y después las pruebas de aceptación, donde se levanta temporalmente la aplicación con Gunicorn para validar que los flujos principales funcionen correctamente desde la perspectiva del usuario. Más adelante, en Upload Test Reports Artifacts, se almacenan los reportes generados durante las pruebas para conservar evidencia de los resultados obtenidos. Luego se ejecuta SonarCloud Scan, que realiza un análisis más profundo sobre la calidad, cobertura y posibles vulnerabilidades del código. Finalmente, cuando el cambio corresponde a un push en la rama principal, se ejecutan los pasos de Docker: Set up QEMU y Set up Docker Buildx preparan el entorno para la construcción de imágenes, Login to Docker Hub autentica el acceso al registro de Docker, y Build and push Docker image construye la imagen de la aplicación y la publica en Docker Hub, dejando la aplicación empaquetada y lista para su despliegue.

## ¿Qué ventajas ofrece empaquetar la aplicación en una imagen Docker al final del pipeline en lugar de simplemente validar el código?

Empaquetar la aplicación en una imagen Docker al final del pipeline ofrece ventajas muy importantes porque deja la aplicación lista para ejecutarse de forma consistente en cualquier entorno. La imagen incluye el código, las dependencias y la configuración necesaria, lo que evita problemas típicos como que algo funcione en desarrollo pero falle en producción. Esto mejora la portabilidad, ya que la misma imagen puede ejecutarse en diferentes servidores sin necesidad de volver a configurar todo desde cero. Además, facilita mucho los despliegues automáticos, porque una vez construida y publicada la imagen, puede usarse directamente en servidores o plataformas como Kubernetes. En otras palabras, no solo se valida que el código esté correcto, sino que también se genera un artefacto listo para ser desplegado de manera rápida y confiable.
